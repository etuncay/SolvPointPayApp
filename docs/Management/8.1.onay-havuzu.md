# 8.1 Onay Havuzu

> Ortak kurallar için bkz. `0.genel-standartlar.md` (özellikle `0.6 Onay Mekanizması`).

## 1. Doküman Künyesi

| Alan | Değer |
|---|---|
| Menü | Operasyonel Süreç Yönetimi › Onay Havuzu |
| Dosya | `8.1.onay-havuzu.md` |
| Ekran tipi | Listeleme + onay/red aksiyonları |

## 2. Amaç ve Kapsam

Maker-checker (1./2. onaycı) gerektiren işlemlerin listelendiği, onaylandığı/reddedildiği merkezi ekrandır.

## 3. Aktörler ve Rol/Yetki Matrisi

| Aktör | Görür | Onaylar | Reddeder | Geri Çeker |
|---|---|---|---|---|
| Başlatan kullanıcı | kendi başlattıkları | — | — | ✓ (kendi) |
| 1. Onaycı | onayına düşenler | ✓ | ✓ | — |
| 2. Onaycı | 1. onay tamamlananlar | ✓ | ✓ | — |

Kullanıcı, 1. Onaycı ve 2. Onaycı **farklı kişiler** olmalıdır.

## 4. Menü ve Navigasyon

Operasyonel Süreç Yönetimi menüsü altında. **Butonlar (filtre):** Onayımı Bekleyenler, Başlattıklarım, Onayladıklarım, Reddettiklerim, Tümü.

## 5. Ekran Yapısı ve Bileşenler

Tek tablo. **Kolonlar:** Referans Numarası, Ekran/İşlem Adı, Başlatan Kullanıcı, İşlem Tarihi, 1. Onaycı, 1. Onay Tarihi, 2. Onaycı, 2. Onay Tarihi, Status (1. Onay Bekleniyor / 2. Onay Bekleniyor / 2. Kademe Reddetti / Onaylandı / Reddedildi).

**Actions:** View, Onayla, Reddet, Geri Çek.

## 6. Alan Listesi

Tüm alanlar sistem tarafından üretilir; kullanıcı doğrudan değiştiremez.

## 7. Validasyon Kuralları

- Reddedilen işlemlerde **yorum zorunlu**; onaylananlarda opsiyonel.
- 2. onaycı için Onayla ikonu, 1. onay tamamlanana kadar pasiftir.

## 8. İş Kuralları ve Akışlar

- Varsayılan filtre **Onayımı Bekleyenler**; İşlem Tarihi'ne göre yeniden eskiye sıralı.
- 2. onaycı, 1. onay tamamlanmamış kayıtları "Onayımı Bekleyenler" filtresinde görmez.
- 2. onay süreci yalnızca 1. onay `Approved` ise başlar.
- Mükerrer aksiyonlarda (çift tıklama) sistem hata döner, statü değişmez.
- Onay ekranı işlem giriş ekranı ile aynıdır; onaycı değişiklik yapamaz (teknik zorluk halinde değiştirilemez düz metin sunulabilir). Değişen değerler vurgulanır (eski/yeni).
- Reddedilen talep başlatan tarafından düzenlenip yeniden gönderilebilir; yeni onay kaydı oluşur, önceki kayıt `Superseded`, `previous_approval_ref` ile ilişkilendirilir.
- **Geri Çek** yalnızca başlatan kullanıcıda görünür; tetiklenince kayıt `Withdrawn` olur, detay düzenlenebilir sunulur. Tekrar Onaya Gönder → `Superseded`; Vazgeç → `Canceled`.

### DB Statü Geçiş Tablosu

| Olay Öncesi DB (1,2) | Event | DB State (1,2) | Önyüz Status |
|---|---|---|---|
| — | İşlem oluşturuldu | Pending, NULL | 1. Onay Bekleniyor |
| Pending, NULL | 1. onaycı Onayladı (1 onaylı) | Approved, NULL | Onaylandı |
| Pending, NULL | 1. onaycı Onayladı (2 onaylı) | Approved, Pending | 2. Onay Bekleniyor |
| Pending, NULL | 1. onaycı Reddetti | Rejected, NULL | Reddedildi |
| Approved, Pending | 2. onaycı Onayladı | Approved, Approved | Onaylandı |
| Approved, Pending | 2. onaycı Reddetti | Approved, Rejected | 2. Kademe Reddetti |
| Approved, NULL | 1. onaycı mükerrer (1 onaylı) | Hata | Onaylandı |
| Approved, Pending | 1. onaycı mükerrer (2 onaylı) | Hata | 2. Onay Bekleniyor |
| Approved, Approved | 2. onaycı mükerrer | Hata | Onaylandı |
| Pending, NULL | 2. onaycı işlem denedi | Hata | 1. Onay Bekleniyor |

## 9. Statü Modeli

Önyüz: 1. Onay Bekleniyor, 2. Onay Bekleniyor, 2. Kademe Reddetti, Onaylandı, Reddedildi. Ek statüler: Superseded, Withdrawn, Canceled.

## 10. Veri Modeli

**`approval_request`**: `id, reference_no, screen_name, payload, initiated_by, initiated_at, first_approver, first_approval_at, first_status, second_approver, second_approval_at, second_status, ui_status, previous_approval_ref, comment`

## 11. Servis / API Sözleşmeleri

| Servis | Method | Açıklama |
|---|---|---|
| `GET /approvals` | GET | Filtreli liste |
| `GET /approvals/{id}` | GET | Onay detayı (giriş ekranı görünümü) |
| `POST /approvals/{id}/approve` | POST | Onayla (idempotent — mükerrerde hata) |
| `POST /approvals/{id}/reject` | POST | Reddet (yorum zorunlu) |
| `POST /approvals/{id}/withdraw` | POST | Geri çek (başlatan) |

## 12. Entegrasyonlar

Yoktur (onay onaylanınca ilgili modül servisini tetikler).

## 13. Yetkilendirme ve Güvenlik

- 1./2. onaycı yetkileri rol matrisinden (`1. Onaycı Olabilir`, `2. Onaycı Olabilir`).
- Onaylanan işlem alt kullanıcılarca değiştirilemeyecek şekilde kilitlenir.

## 14. Loglama ve Denetim İzi

- Tüm onay/red/geri çek aksiyonları kullanıcı + zaman + yorum ile loglanır.

## 15. Hata Yönetimi

- Mükerrer onay aksiyonunda hata döner, statü değişmez.

## 16. Bildirimler

- Onay bekleyen, onaylanan, reddedilen işlemler için ilgili kullanıcılara bildirim (parametrik).

## 17. Batch / Zamanlanmış İşler

Yoktur.

## 18. Bağımlılıklar

Onay gerektiren tüm ekranlar; Onay Kuralları (12.2), Roller (12.3).

## 19. Açık Sorular ve Riskler

- Onay ekranının "giriş ekranı ile aynı" gösterilmesi tüm ekran tipleri için teknik olarak mümkün olmayabilir; düz metin fallback kapsamı netleştirilmeli.

## 20. Kabul Kriterleri ve Test Senaryoları

- [ ] DB statü geçiş tablosundaki tüm senaryolar doğru çalışır.
- [ ] 2. onaycı 1. onay tamamlanmadan onaylayamaz.
- [ ] Mükerrer onay hata döner.
- [ ] Reddedilen talep yeniden gönderilince eski kayıt Superseded olur.
- [ ] Geri Çek yalnızca başlatanda görünür.
