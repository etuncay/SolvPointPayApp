# 1. Ana Sayfa — Dashboard

> Ortak kurallar için bkz. `0.genel-standartlar.md`. Bu doküman yalnızca menüye özel davranışları içerir.

## 1. Doküman Künyesi

| Alan | Değer |
|---|---|
| Menü | Ana Sayfa / Dashboard |
| Dosya | `1.ana-sayfa-dashboard.md` |
| Uygulama | BackOffice |
| Ekran tipi | Dashboard (widget tabanlı, salt-okunur) + Ayarlar |
| Bağımlılık | Tüm operasyonel modüller (özet veri kaynağı) |

## 2. Amaç ve Kapsam

Kullanıcının role göre özelleştirilmiş; bekleyen işleri, günlük operasyonel metrikleri ve sistem sağlığını tek ekranda gördüğü açılış sayfasıdır. Ekran **salt görüntülemedir**; üzerinde veri değişikliği yapılmaz, widget'lar ilgili modüllere yönlendirir.

Kapsam: Dashboard widget altyapısı + sağ üstte yer alan **Ayarlar** ekranı. İlk fazda örnek widget'lar **canlıya alınmaz**; yalnızca widget çatısı (framework) kurulur.

## 3. Aktörler ve Rol/Yetki Matrisi

| Aktör | Yetki |
|---|---|
| Tüm BackOffice kullanıcıları | Dashboard'u görür; widget görünürlüğü ve detay seviyesi **role göre** değişir |
| Operasyon | Bekleyen transfer, KYC, AML widget'ları |
| Finans/Muhasebe | Finansal özet, mutabakat widget'ları |
| Uyum | AML/KYC/vaka widget'ları |
| Yönetim | Tüm metrik ve özet grafikler |

Dashboard yetkisi `List` bazlıdır (Insert/Update/Delete yoktur). Her widget için ayrı **rol bazlı görünürlük** tanımı bulunur.

## 4. Menü ve Navigasyon

- Menüde en üstte yer alır; login sonrası ilk açılan ekrandır.
- **Ayarlar**: sağ üst köşede ikon olarak yer alır, sol menüde **listelenmez**.
- Widget'lara tıklanınca ilgili kaynak ekrana (filtreli) yönlendirme yapılır.

## 5. Ekran Yapısı ve Bileşenler

### 5.1 Dashboard

Widget tabanlı grid yerleşim. Her widget rol bazlı görünür/gizlenir. Tablo tipli widget'larda ilk 10 satır gösterilir; "tam ekran" seçeneği ile sıralama/filtreleme açılır.

**Bekleyen iş widget'ları birbirini dışlar** — bir işlem yalnızca tek widget'ta görünür (örn. AML'e takılan transfer "Bekleyen Transferler"de görünmez, yalnızca "AML Kontrollerine Takılan Transferler"de).

Örnek widget seti (ilk fazda yalnızca çatı kurulur, içerik canlı değil):

- Onayımı Bekleyen İşler
- Bekleyen Transferler
- KYC Manuel Doğrulama Talepleri
- AML Kontrollerine Takılan Transferler
- Reddedilen / İptal Edilen İşlemler
- Başarılı/Başarısız Günlük İşlem Adet ve Tutar Grafikleri
- Günlük Yeni Müşteri Adedi Grafiği
- Gün içi en fazla para gönderen / alan / çeken 10 müşteri
- Gün içi en fazla para alan / veren 10 temsilci
- Sistem Hataları (başarılı istek oranı, en çok hata üreten 3 servis, p95/p99 latency — kritik 3 endpoint)

### 5.2 Ayarlar (sağ üst, menüde yok)

- **Parola**: Eski Parola, Yeni Parola, Yeni Parola (Tekrar), Güncelleme Sıklığı (combobox: 1 / 3 / 6 ay)
- **Karşılama Mesajı**: kullanıcıya özgü phishing önlemi metni
- **Uygulama Ayarları**: Dil Tercihi, Tema (Açık/Koyu), Metin Boyutu (Küçük/Standard/Büyük/Ekstra Büyük)
- **Hatalı Girişler**: son başarısız erişim teşebbüsleri listesi

## 6. Alan Listesi

| Alan | Tip | Kaynak | Açıklama |
|---|---|---|---|
| Widget verileri | hesaplanmış | İlgili modül servisleri | Salt-okunur, periyodik yenilenir |
| Eski/Yeni/Tekrar Parola | string | Kullanıcı girişi | Parola değişikliği |
| Güncelleme Sıklığı | enum | Kullanıcı girişi | 1/3/6 ay |
| Karşılama Mesajı | string | Kullanıcı girişi | — |
| Dil / Tema / Metin Boyutu | enum | Kullanıcı girişi | UI tercihleri |

## 7. Validasyon Kuralları

- Yeni Parola = Yeni Parola (Tekrar) eşleşmeli.
- Yeni Parola kurum parola politikasına uymalı (uzunluk, karmaşıklık); eski parola ile aynı olamaz.
- Karşılama Mesajı: regex pattern kontrolü (XSS/script engelleme), max uzunluk.

## 8. İş Kuralları ve Akışlar

- Dashboard metrik tutarları **TRY** cinsinden; çoklu para birimi işlem-anı kuru ile çevrilir.
- Günlük metrikler İstanbul saat dilimi takvim gününe göre hesaplanır.
- Bekleyen iş widget'ları **mutually exclusive**: kümeler kesişmez.
- Parola değişikliği başarılı olduğunda kullanıcı yeniden login'e yönlendirilir (mevcut oturum revoke).
- Son başarılı girişten sonra başarısız erişim teşebbüsü varsa kullanıcıya bilgilendirme mesajı gösterilir.

## 9. Statü Modeli

Bu ekrana özgü statü modeli yoktur.

## 10. Veri Modeli

Dashboard kendi tablosunu tutmaz; verileri ilgili modüllerden okur. Kullanıcı tercihleri için:

**`user_preference`**: `id, user_id, language, theme, text_size, welcome_message, password_change_frequency, updated_at, updated_by`

**`dashboard_widget_visibility`**: `id, widget_code, role_id, is_visible, detail_level`

## 11. Servis / API Sözleşmeleri

| Servis | Method | Açıklama |
|---|---|---|
| `GET /dashboard/widgets` | GET | Kullanıcının rolüne göre görünür widget listesi + veri |
| `GET /dashboard/widget/{code}` | GET | Tek widget verisi (lazy load) |
| `PUT /user/preferences` | PUT | UI tercihleri günceller |
| `POST /user/change-password` | POST | Parola değiştirir (idempotent değil; oturum revoke) |
| `GET /user/failed-logins` | GET | Hatalı giriş listesi |

## 12. Entegrasyonlar

Doğrudan dış entegrasyon yoktur. Sistem Hataları widget'ı için APM/monitoring kaynağından (latency, hata oranı) veri okunur.

## 13. Yetkilendirme ve Güvenlik

- Her widget rol bazlı görünürlük denetiminden geçer (backend enforce).
- Parola değişikliği mevcut oturumu sonlandırır; tüm refresh token'lar revoke edilir.

## 14. Loglama ve Denetim İzi

- Parola değişikliği loglanır (yalnızca "değişiklik yapıldı"; parola loglanmaz).
- UI tercih değişiklikleri loglanır.
- Dashboard erişimi başarılı/başarısız erişim logu kapsamındadır.

## 15. Hata Yönetimi

- Widget veri servisi hata verirse o widget "veri yüklenemedi" göstergesiyle gelir, sayfa bütünlüğü bozulmaz.
- Parola politikası ihlali kullanıcıya net mesajla bildirilir.

## 16. Bildirimler

- Parola değişikliğinde kullanıcıya bilgilendirme (e-posta) gönderilir.

## 17. Batch / Zamanlanmış İşler

Dashboard metrikleri için ağır sorgular periyodik (örn. 5–15 dk) önbelleğe alınabilir; "Sistem Hataları" widget'ı near-real-time okunur.

## 18. Bağımlılıklar

Para Transferleri, Müşteriler, Temsilciler, Risk ve Uyum, Onay Havuzu, KYC Yönetimi, Sistem Yönetimi (rol/parametre).

## 19. Açık Sorular ve Riskler

- Örnek widget seti ilk fazda canlı değildir; **hangi widget'ların ilk sürümde aktif olacağı** netleştirilmelidir.
- Widget veri tazeleme periyodu parametrik mi olacak, belirsiz.
- "Detay seviyesi" (rol bazlı) tanımının somut anlamı netleştirilmeli.

## 20. Kabul Kriterleri ve Test Senaryoları

- [ ] Dashboard role göre yalnızca yetkili widget'ları gösterir.
- [ ] Bir bekleyen iş kalemi yalnızca tek widget'ta görünür (kesişim yok).
- [ ] Tüm tutarlar TRY cinsinden ve doğru kurla gösterilir.
- [ ] Tablo widget ilk 10 satırı gösterir; tam ekranda sıralama/filtre çalışır.
- [ ] Parola değişikliğinde eşleşme/politika kontrolleri çalışır, başarıda oturum sonlanır.
- [ ] Hatalı giriş listesi son başarısız teşebbüsleri doğru gösterir.
- [ ] Bir widget servisi hata verdiğinde sayfa çökmez.
